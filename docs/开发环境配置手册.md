### 虚拟环境是什么

把 Python 想象成一个公共厨房，全局环境就是公共灶台，所有人共用。问题是：项目 A 需要 `pandas 1.x`，项目 B 需要 `pandas 2.x`，装在一起必然冲突。

虚拟环境（`.venv`）就是给每个项目单独隔出一个小厨房，里面的锅碗瓢盆互不干扰。

```
全局 Python（公共厨房）
├── bridge-check/.venv/    ← 项目 A 的独立环境
├── report-tool/.venv/     ← 项目 B 的独立环境
└── data-check/.venv/      ← 项目 C 的独立环境
```

**为什么必须每个项目一个？**

假设你有项目 A 和项目 B 都装到全局环境，A 需要 `numpy 1.20`，B 升级到 `numpy 2.0`。结果就是：你为 B 装新版 numpy，A 突然就跑不起来了——而你完全不知道是因为这次升级搞坏的。十几个项目共用全局环境，一两个月就会变成”我也不知道哪儿出问题了”的死局。

**虚拟环境本质就是把库装进项目文件夹的 `.venv/` 里**，每个项目库版本独立，互不干扰。

### Git 是什么

Git 是代码的**时光机**，每次 `commit` 就是存一个存档点，随时可以回到任意历史版本。GitHub 是存档的**云端保险箱**，本地电脑坏了，云端还有。

三个核心区域：

```
工作区（你写代码的地方）
    ↓ git add（选）
暂存区（准备提交的文件待在这里）
    ↓ git commit（存档）
本地仓库（存档在本地）
    ↓ git push（上传）
远程仓库（GitHub 上的云端备份）
```

**为什么要分三步？**

- **`git add`（选）**：你今天改了 5 个文件，但只有 3 个改完了，另外 2 个还是半成品，可以只 add 这 3 个，其余暂时不进入存档
- **`git commit`（存）**：本地存档，写一句话备注”这次改了什么”，方便日后回看
- **`git push`（传）**：把本地存档同步到 GitHub 云端

如果一步搞定，会变成”任何改动都立刻全部上传”，没有筛选、没有备注、没有缓冲。三步分离让你能精确控制每次提交的内容。

### VS Code、.py 文件、.bat 文件的关系

- **VS Code（车间）**：在这里写代码、调试、测试
- **.py 文件（发动机）**：真正执行任务的核心代码
- **.bat 文件（启动面板）**：代码写完调试好后，双击直接运行，不需要打开 VS Code

### 路径铁律

> ⚠️ **桌面建项目是最常见的新手错误，必须从一开始就杜绝。**

桌面的真实路径是 `C:\Users\张德帅\Desktop\`，含中文、含空格，双重违规。

- ❌ `C:\Users\张德帅\桌面\我的项目\`  → 中文路径，必出问题
- ❌ `C:\Users\张德帅\Desktop\bridge check\`  → 含空格，必出问题
- ❌ `D:\检测项目\2024年\桥梁数据\`  → 中文路径，必出问题
- ✅ `D:\CodeProjects\bridge-check\`  → 纯英文，无空格，正确

**为什么这么严格？**
很多 Python 库（尤其是 Office 处理库）在路径含中文或空格时会报奇怪的编码错误，排查极难。直接禁用这两类字符可以省掉 90% 的玄学问题。

### 项目命名规范

文件夹名和 GitHub 仓库名应该保持一致，遵循以下规则：

- **小写字母**，单词之间用**连字符** `-` 分隔（kebab-case）
- **见名知意**，避免 `test`、`new`、`xiangmu1`、`untitled` 这种无意义命名
- 工具类项目可加后缀：`-tool`、`-cli`、`-gui`

|❌ 不推荐         |✅ 推荐                     |说明        |
|--------------|-------------------------|----------|
|`test1`       |`bridge-detection-report`|见名知意      |
|`MyProject`   |`my-project`             |全小写更通用    |
|`bridge_check`|`bridge-check`           |连字符比下划线更主流|
|`项目1`         |`road-survey-2024`       |不用中文      |


> 💡 当你有十几个项目时，规范的命名能让你 3 秒钟找到目标，而不是一个个点开看。

### Git Bash 里的路径斜杠

Windows 系统用反斜杠 `\`，但 **Git Bash 用正斜杠 `/`**：

```bash
# ❌ Git Bash 里反斜杠会报错
cd D:\CodeProjects

# ✅ Git Bash 里用正斜杠
cd D:/CodeProjects
```

> 在 VS Code 的 Git Bash 终端里写命令，路径一律用 `/`。Python 代码里也建议统一用 `/`，跨平台兼容。


> 💡 **Git Bash 粘贴技巧：** 在 Git Bash 终端中，`Ctrl+V` 是特殊控制符无法粘贴，粘贴请使用 **`Shift+Insert`** 或点击鼠标右键。

### 相对路径 vs 绝对路径

代码里引用文件时，**绝对路径**写死了电脑位置，换电脑就报错：

```python
# ❌ 绝对路径：只在你自己电脑上能跑
df = pd.read_excel("C:\\Users\\张德帅\\Desktop\\data.xlsx")
```

**相对路径**以项目文件夹为起点，换电脑也能用：

```python
# ✅ 相对路径：./data/data.xlsx 表示项目文件夹内的 data 子文件夹
df = pd.read_excel("./data/data.xlsx")
```

推荐的项目文件结构：

```
my-project/
├── .venv/                  ← 虚拟环境（自动生成，不要上传）
├── .python-version         ← 锁定 Python 版本
├── pyproject.toml          ← 核心配置文件
├── README.md
└── src/                    ← ！！！所有业务代码必须放在这里！！！
    └── my_project/         
        ├── __init__.py     ← 标识这是一个标准的 Python 包
        └── py.typed        ← 声明支持静态类型推导
```

> 养成用相对路径的习惯，代码上传 GitHub 后别人克隆也能直接运行。

### 上传 vs 忽略 速查表

|文件 / 文件夹               |操作        |原因                   |
|-----------------------|----------|---------------------|
|`*.py` 所有代码文件          |✅ **必须上传**|这就是项目本体              |
|`pyproject.toml`       |✅ **必须上传**|uv 项目配置文件            |
|`uv.lock`              |✅ **必须上传**|锁定依赖版本，保证重现一致        |
|`.python-version`      |✅ **必须上传**|指定 Python 版本         |
|`.gitignore`           |✅ **必须上传**|告诉 Git 忽略什么          |
|`README.md`            |✅ **建议上传**|项目说明，GitHub 首页显示     |
|`.env.example`         |✅ **建议上传**|配置模板，不含真实密码          |
|`*.bat` 启动脚本           |✅ **建议上传**|换电脑后直接用              |
|`.venv/`               |❌ **必须忽略**|体积巨大，每台电脑重新生成        |
|`__pycache__/`         |❌ **必须忽略**|Python 自动生成的缓存，无意义   |
|`.env`                 |❌ **必须忽略**|含密码/密钥，泄露即事故         |
|`~$*.xlsx` `~$*.docx`  |❌ **必须忽略**|WPS/Office 打开时的临时锁定文件|
|`.DS_Store` `Thumbs.db`|❌ **必须忽略**|Mac/Windows 系统垃圾文件   |
|大体积数据文件 `*.csv` `*.pdf`|❌ **建议忽略**|GitHub 单文件限制 100MB   |

## 三、Python 引擎：安装 uv 包管理器（全局安装一次）

### 前置条件：解除 PowerShell 脚本执行限制（新电脑必做）

```powershell
Set-ExecutionPolicy RemoteSigned -scope CurrentUser
```

**为什么要这么做？**

Windows 出于安全考虑，**默认禁止 PowerShell 执行任何 `.ps1` 脚本**——防止有人骗你下载并双击恶意脚本。但虚拟环境激活、uv 安装等操作都需要执行 `.ps1` 脚本。

`RemoteSigned` 是一个折中策略：

- ✅ **本地写的脚本**可以直接运行（你自己的虚拟环境激活脚本）
- ✅ **网络下载的脚本**必须经过数字签名才能运行（防止恶意脚本）

`-scope CurrentUser` 表示只对当前用户生效，不影响其他用户，更安全。

> 不执行这步的后果：激活 `.venv` 时报错 `无法加载文件，因为在此系统上禁止运行脚本`。

### 推荐方式：官方独立安装

不依赖系统 Python，最干净的方式。在 PowerShell 中执行：

```powershell
powershell -ExecutionPolicy ByPass -c "irm https://astral.sh/uv/install.ps1 | iex"
```

安装完成后**重启终端**，验证：

```bash
uv --version
```

### 备选方式：通过 pip 安装

如果你的电脑已经有 Python 和 pip：

```bash
pip install uv
```



### 安装指定版本的 Python

```bash
uv python install 3.12
```

uv 会下载并安装 Python 3.12，不需要去官网下载，不需要配环境变量。

### 查看已安装的 Python 版本

```bash
uv python list
```

### 在项目里指定 Python 版本

进入项目目录后执行：

```bash
uv python pin 3.12
```

会生成一个 `.python-version` 文件，记录这个项目用 3.12。这个文件需要上传到 GitHub，别人克隆后会自动用同一版本。

### 同时存在多个 Python 版本

uv 会自动选择项目指定的版本，不需要手动切换系统默认。哪怕系统装着 3.11、3.13、3.14，每个项目都能用自己的版本互不干扰。

## 四、Git 与 GitHub 本地联动配置（每台电脑执行一次）

### SSH 密钥是什么、为什么要用

SSH 密钥是一对配对的文件：

- **私钥**（`id_ed25519`）：留在你的电脑上，**绝对不能给任何人**
- **公钥**（`id_ed25519.pub`）：上传到 GitHub，可以公开

**工作原理**（不严格但够用的比喻）：
公钥像一把**挂在 GitHub 上的锁**，私钥是**留在你电脑里的钥匙**。每次推送代码时，Git 会自动用本地私钥去开 GitHub 上的锁，开得开就放行，开不开就拒绝。

**为什么不用密码？**

- 密码每次推送都要输，烦
- 密码可能被偷看、被猜中
- SSH 密钥靠数学算法验证，比密码安全得多，且免输

### 1. 先检查本机是否已有 SSH 密钥

> ⚠️ GitHub 官方建议：生成新密钥之前先检查本机有没有现成的，避免重复生成覆盖掉旧密钥。

在 Git Bash 中执行：

```bash
ls -al ~/.ssh
```

**如何判断已有密钥？** 看到**成对存在**的两个文件就是了：

```
id_ed25519        ← 私钥（无后缀）
id_ed25519.pub    ← 公钥（带 .pub 后缀）
```

如果两个都有 → **跳过第 2 步**，直接从第 3 步开始把 `.pub` 公钥添加到 GitHub。

如果文件夹是空的或提示 `No such file or directory` → 按第 2 步生成新密钥。

### 2. 生成新密钥

```bash
ssh-keygen -t ed25519 -C "your_email@example.com"
```

连续按三次回车，保持默认路径，不设密码。

> ⚠️ Windows 用户名为中文时，密钥在 `C:\Users\中文名\.ssh`，新版 Git 通常能识别，遇到报错再处理。

### 3. 获取公钥

```bash
cat ~/.ssh/id_ed25519.pub
```

复制输出的完整内容（以 `ssh-ed25519` 开头的一整行）。

> ⚠️ **复制的是 `.pub` 公钥文件内容，不是无后缀的私钥**！如果误把私钥贴到 GitHub，等于把家门钥匙挂到了网上。

### 4. 添加到 GitHub

GitHub → 头像 → **Settings** → **SSH and GPG keys** → **New SSH key**

- Title：备注这台电脑（如 `Office-PC`、`Home-PC`）
- Key：粘贴公钥

> 多台电脑各添加一个 SSH Key，互不影响。

### 5. 测试连接

```bash
ssh -T git@github.com
```

> ⚠️ **首次连接必踩的坑：**
> 执行后会出现以下提示：
> 
> ```
> The authenticity of host 'github.com' can't be established.
> Are you sure you want to continue connecting (yes/no/[fingerprint])?
> ```
> 
> **必须手动输入 `yes` 并回车**，不能直接按回车，否则连接会被拒绝。
> 这个提示只会出现一次，之后不再询问。

看到 `Hi 用户名! You've successfully authenticated` 即成功。

---

```bash
git config --global user.name "你的GitHub用户名"
git config --global user.email "你的GitHub邮箱"
```

这是告诉 Git “提交人是谁”，每次 commit 会自动署名。`--global` 表示对所有项目生效，配一次就够。

## 阶段二：实战——从零跑通第一个项目

> 带新手走完一个完整的闭环。


## 五、新建项目完整流程（推荐：uv 项目化工作流）

> ✅ **官方推荐使用 `uv init` + `uv add` + `uv sync` + `uv run` 现代化工作流**，靠 `pyproject.toml` 和 `uv.lock` 管理依赖，比传统的 `requirements.txt` 更可靠。

### pyproject.toml 和 uv.lock 的区别

这两个文件都要上传，但作用不同：

|文件              |作用                |谁来写           |例子                                 |
|----------------|------------------|--------------|-----------------------------------|
|`pyproject.toml`|**声明需要哪些库**（人读）   |你（通过 `uv add`）|`pandas, openpyxl, python-docx`    |
|`uv.lock`       |**锁定具体到补丁版本**（机器读）|uv 自动生成       |`pandas==2.1.4, openpyxl==3.1.2...`|

**为什么要两个文件？**

`pyproject.toml` 只说”我需要 pandas”，不限定版本。如果只靠它还原环境，A 电脑装到 `pandas 2.1`，半年后 B 电脑装就可能是 `pandas 2.5`，可能引入兼容问题。

`uv.lock` 把所有库（包括库的库）的精确版本号锁死。`uv sync` 时严格按 lock 文件还原，保证所有电脑环境**一字不差**。

### 流程总览

```
第 1 步  D:/CodeProjects/ 下进入纯英文路径（绝对不在桌面）
第 2 步  uv init 初始化项目（自动生成项目骨架）
第 3 步  VS Code 打开文件夹
第 4 步  uv add 添加所需库（自动创建 .venv 并锁定版本）
第 5 步  VS Code 选择 Python 解释器
第 6 步  代码质量基建与自动化规范
第 7 步  完善 .gitignore
第 8 步  完善 README.md
第 9 步  在 GitHub 新建仓库（注意 Private/Public 选择）
第 10 步  git init → add → commit → push
```

### 第 1 步：进入项目目录

确保路径纯英文、无空格，建议在 D 盘新建 CodeProjects 文件夹。

终端运行下列代码进入：

```bash
cd D:/CodeProjects
```



### 第 2 步：初始化项目

> ⚠️ **新手必看**：在敲下初始化命令前，你必须决定这个项目的“体量”。架构选错，后期极易引发模块导入混乱的玄学报错。

根据项目规划，这里提供两种初始化模式。**强烈建议业务类、工具类项目一律采用模式二。**

#### 模式一：扁平布局（Flat Layout）
适合：**一次性的几十行数据处理脚本**、不需要打包分享的自用小工具。

终端运行：

> 💡提示：bridge-check 是示例的项目文件夹名，可按需替换。

```bash
uv init bridge-check
cd bridge-check
```

执行后 uv 会自动生成以下文件：

```plaintext
bridge-check/
├── .python-version       ← 自动指定 Python 版本
├── .gitignore            ← uv 已自动写入基础规则
├── README.md             ← 项目说明模板
├── main.py               ← 示例入口文件
└── pyproject.toml        ← 项目配置（依赖记录在这里）
```

- 优点：极其简单，随手写完 uv run main.py 就能跑。

- 致命缺点：所有代码、测试、配置文件全堆在一起。当文件变多时，极易与 Python 系统环境里的第三方库发生“命名冲突”（例如你建了个 email.py，会导致整个环境的邮件库瘫痪）。

#### 模式二：工程布局（src-layout）—— 🌟 强烈推荐
适合：需要长期维护的项目、包含多个自定义模块、未来准备编译打包成 .exe 桌面软件的工程。

终端运行：

```bash
uv init --lib bridge-check
cd bridge-check
```

生成结构：

```plaintext
bridge-check/
├── .python-version
├── pyproject.toml        ← 项目配置（自动配置了打包构建系统）
└── src/                  ← ！！！核心变化在这里！！！
    └── bridge_check/     ← 你的专属代码包（业务逻辑全放这里）
        ├── __init__.py   ← 声明这是一个正规的 Python 包
        └── py.typed
```
- **优点 1（物理隔离）：** 代码在 src/ 下，配置文件在根目录，清晰解耦。

- **优点 2（杜绝导包灾难）：** 强制你使用绝对路径导入（例如 from bridge_check.utils import reader），彻底解决当前目录文件与第三方库重名导致的玄学报错。

- **优点 3（测试与分发）：** 这是 Python 业界最推崇的标准格式，完美兼容未来的单元测试（pytest）和二进制编译（PyInstaller）。

>💡 后续操作提醒：
>如果你选择了**模式二**，uv 不会自动生成 main.py。你需要手动在 src/bridge_check/ 目录下新建 main.py 作为程序入口。

> 💡 这些文件除 `.venv/`（暂时还没生成）之外都需要上传到 GitHub。

### 第 3 步：VS Code 打开文件夹

终端运行：

```bash
code .
```


> ⚠️ **首次打开必弹窗：**
> 新手第一次使用 `code .` 打开文件夹时，VS Code 会弹出提示：**"是否信任此工作区中的作者？(Do you trust the authors...)"**
> 必须勾选并点击 **"是，我信任此作者"**，否则代码无法运行且很多功能受限。很多新手因为害怕而点否，导致后续调试报错。


### 第 4 步：添加项目依赖

根据需要安装库，以下是处理 Office 文件的常用组合：

```bash
uv add python-docx openpyxl pandas pypdf python-dotenv
```

|库名             |用途            |
|---------------|--------------|
|`python-docx`  |读写 Word 文档    |
|`openpyxl`     |读写 Excel 文件   |
|`pandas`       |表格数据处理分析      |
|`pypdf`        |读取 PDF 文件     |
|`python-dotenv`|读取 `.env` 配置文件|


> ✅ **第一次执行 `uv add` 时 uv 会自动做几件事**：
> 
> 1. 检查并下载安装项目需要的 Python 版本（按 `.python-version` 指定的，如果还没装）
> 2. 创建 `.venv` 虚拟环境
> 3. 把依赖记录到 `pyproject.toml`
> 4. 生成或更新 `uv.lock` 锁定具体版本号
> 
> 这一条命令搞定一切，不需要再手动 `uv venv` 或 `uv pip freeze`。

### 第 5 步：选择正确的 Python 解释器（每个项目必做）

`Ctrl+Shift+P` → `Python: Select Interpreter` → 选带 `.venv` 字样的那个。

**为什么必须做这一步？**

VS Code 里可能同时存在多个 Python：系统全局的、各个项目的 `.venv`、其他工具自带的。VS Code 不知道你想用哪个跑代码，需要你**手动指定**。

选了正确的 `.venv` 之后，VS Code 才知道：

- 跑 `python xxx.py` 时该用哪个 Python
- 代码里 `import pandas` 时去哪个文件夹找库
- 智能提示从哪里读取库的函数列表

> ⚠️ 没选对的典型现象：明明 `uv add pandas` 装好了，VS Code 却在 `import pandas` 下画红线提示”找不到模块”——其实是用错 Python 了。

### 第 6 步：代码质量基建与自动化规范

在多人协作或长期维护的项目中，代码风格不一致（空格乱敲、变量乱命名、无用导入）会导致极高的维护成本。靠人工检查是徒劳的，必须通过底层工具进行强制约束，做到**严禁带红线（报错和警告）提交**。

#### 1. 区分“运行依赖”与“开发依赖”
在安装代码检查工具之前，必须先理解依赖的分级概念：

- 运行依赖（标准 add）：例如 pandas, pyside6。这些是程序运行不可或缺的库。未来打包成 .exe 时，这些库必须被一起打包。

 - 开发依赖（加 --dev 参数）：例如 ruff（代码检查）, pytest（单元测试）。这些只是你在开发阶段用来辅助你的“施工工具”，程序真正交付运行或打包时，完全不需要它们。

操作命令：

在项目根目录（VS Code 终端）执行以下命令，专门安装开发工具：

```bash
uv add --dev ruff pytest
```

> 💡 底层变化： 执行后打开 pyproject.toml，你会发现这些库被单独记录在了 [tool.uv] 或 [project.optional-dependencies] 相关的开发依赖区块中，实现了工具与业务的彻底物理隔离。
#### 2. 在 VS Code 中配置 Ruff 自动化

我们需要让编辑器和 Ruff 联动，实现“按下保存键，代码自动排版并修复低级错误”。

- 第一步：安装插件
    
    在 VS Code 插件市场搜索并安装官方插件：Ruff (作者：Astral Software)。

- 第二步：更新工作区设置

    按 `Ctrl+Shift+P` → 输入 `Open User Settings JSON`，将之前的配置升级为以下内容：

```json
{
  "editor.fontSize": 14,
  "editor.tabSize": 4,
  "editor.wordWrap": "on",
  "files.autoSave": "afterDelay",
  "files.autoSaveDelay": 1000,
  "terminal.integrated.defaultProfile.windows": "Git Bash",
  "python.terminal.activateEnvironment": true,
  
  // ====== 以下为新增的 Ruff 自动化配置 ======
  
  // 将 Python 文件的默认格式化工具指定为 Ruff
  "[python]": {
    "editor.defaultFormatter": "charliermarsh.ruff",
    "editor.formatOnSave": true,         // 保存时自动格式化排版
    "editor.codeActionsOnSave": {
      "source.fixAll": "explicit",       // 保存时自动修复可修复的警告（如删除无用的 import）
      "source.organizeImports": "explicit" // 保存时自动按字母顺序梳理顶部的 import
    }
  }
}
```

#### 3. 定义项目的“红线规则”

不同项目对代码的严格程度要求不同。我们需要在项目的配置文件中立下规矩。

打开项目根目录的 `pyproject.toml` 文件，在最底部追加以下内容：

```ini, TOML
# ====== Ruff 规则配置区块 ======
[tool.ruff]
# 设定代码一行的最大长度。超过此长度 Ruff 会自动帮你换行，避免横向滚动条过长
line-length = 100 
# 设定需要检查的源代码目录，排除虚拟环境等无关文件
src = ["src", "tests"]

[tool.ruff.lint]
# 选择要启用的检查规则集
select = [
    "E",  # pycodestyle 错误（如缩进、空格不规范）
    "F",  # Pyflakes（如定义了变量却没使用、引入了没用的库）
    "I",  # isort（import 排序规范）
    "W",  # pycodestyle 警告
]
# 忽略某些特定的规则（如果你觉得某条规则太严厉，可以把它的编号写在这里）
#例如可修改为 ignore = ["RUF001", "RUF002", "RUF003"]（明确忽略易混淆字符（中文标点）规则，防止误报）
ignore = []
```
#### 4. 日常开发纪律
完成以上配置后，你的开发体验会发生质的改变：

1. **自动清理**：如果你引入了 `import os` 但整篇代码都没用到，一按保存（`Ctrl+S`），这行代码会被 Ruff 瞬间自动删掉。

2. **强制对齐**：随意打乱的函数参数缩进，保存瞬间会自动排列整齐。

3. **红线拦截**：VS Code 编辑器里如果出现红色或黄色波浪线，必须当场解决，绝对不允许带着波浪线执行 git push 上传代码。

## 六、.gitignore 完整配置

### .gitignore 工作原理（先理解再写规则）

`.gitignore` 是一份”**这些文件 Git 别管**“的清单。但有个关键限制：

> ⚠️ **`.gitignore` 只对”还没被 Git 追踪过”的文件生效。**

也就是说：

- ✅ **新文件**：写进 `.gitignore` 后 Git 直接无视，不会出现在 `git status` 里
- ❌ **已经 commit 过的文件**：再加进 `.gitignore` 没有任何效果，Git 仍然会追踪它

这就是为什么强调要在**第一次 `git add` 之前**就写好 `.gitignore`。事后补救需要先用 `git rm --cached` 把文件从追踪列表中移除（见后文）。

### 完整 .gitignore 模板

在项目根目录新建 `.gitignore` 文件（以点开头，无后缀），写入以下内容：

```gitignore
# ==========================================
# ✅ 以下文件不在此列表 = 会被上传到 GitHub
# pyproject.toml    → 必须上传（项目配置）
# uv.lock           → 必须上传（依赖版本锁定）
# .python-version   → 必须上传（Python 版本指定）
# .gitignore        → 必须上传
# README.md         → 必须上传
# .env.example      → 建议上传（配置模板）
# *.bat             → 建议上传 （启动脚本）
# 所有 .py 文件     → 必须上传
# ==========================================


# ===== ❌ Python 虚拟环境（体积大，必须忽略）=====
# .venv 是 uv 创建的虚拟环境文件夹，里面是按项目装的所有库
# 体积可达几百 MB，且每台电脑应该自己重新生成（uv sync）
.venv/
venv/
__pycache__/        # Python 运行时自动生成的字节码缓存
*.py[cod]           # 编译后的 Python 文件
*.pyo
*.egg-info/
dist/
build/
.eggs/

# ===== ❌ 敏感配置（含密码/密钥，绝对不能上传）=====
.env
.env.local
.env.production
config.ini
secrets.json
*.key
*.pem
*.p12

# ===== ❌ 系统垃圾文件 =====
.DS_Store           # Mac 系统在每个文件夹自动生成的隐藏文件
Thumbs.db           # Windows 缩略图缓存
desktop.ini         # Windows 文件夹自定义图标配置

# ===== ❌ VS Code 本地配置 =====
.vscode/
*.code-workspace

# ===== ❌ 日志与临时文件 =====
*.log
*.tmp
*.temp
/temp/
/cache/
/output/

# ===== ❌ WPS / Office 临时锁定文件 =====
# 当你在 WPS 或 Office 打开 report.docx 时，
# 系统会同时生成一个 ~$report.docx 隐藏文件，
# 用来标记"这个文档正在被编辑"。
# 关闭文档后通常会自动删除，但偶尔会残留。
# 这些临时文件不是你的内容，不应该上传。
~$*.docx
~$*.xlsx
~$*.pptx
~$*.doc
~$*.xls

# ===== ❌ 大体积数据文件（GitHub 单文件限制 100MB）=====
# 按需取消注释：
# *.csv
# *.xlsx
# *.pdf
# /data/
```

### 验证 .gitignore 是否生效

写完规则后执行：

```bash
git status
```

**看不到的文件就是被忽略成功的。** 例如 `.venv/` 没出现在列表里，说明规则生效。

### .env 和 .env.example 配合使用

项目里如果有数据库密码、API 密钥等敏感信息，正确做法是：

**第一步：建 `.env` 存真实内容（加入 .gitignore，不上传）**

```

```

**第二步：建 `.env.example` 存模板（上传到 GitHub）**

```

```

**第三步：安装读取库并在代码里调用**

```bash
uv add python-dotenv
```

```python
from dotenv import load_dotenv

password = os.getenv("DB_PASSWORD")
```

> 别人克隆项目后复制 `.env.example` 改名为 `.env`，填入自己的密钥即可运行。

- 关掉终端不需要 `deactivate`

## 九、代码排错与调试基础（代码报错的救命技能）

调试比 `print()` 排错强 100 倍，可以让代码停在指定行，逐行查看每个变量的值。

### 打断点

在代码行号**左侧灰色区域**点击鼠标，会出现一个**红点**，这就是断点。代码执行到这一行会自动暂停。

```python
def calculate(data):
    total = 0
    for item in data:
        total += item.value     # ← 在这一行号左边点一下，出现红点
    return total
```

### 启动调试

按 `F5` 启动调试。第一次会让选择”Python 文件”，选这个就行。

代码会运行到红点处停下，进入调试视图。

### 查看变量

调试模式下，VS Code 左侧自动显示**变量面板**，列出当前所有变量的值：

```
VARIABLES
  ▼ Locals
      total: 0
      data: [<Item>, <Item>, ...]
      item: <Item object at 0x...>
```

把鼠标悬停在代码里任何变量名上，也能直接看到当前值。

### 控制执行

代码顶部会出现调试工具栏：

|按钮    |快捷键        |功能         |
|------|-----------|-----------|
|▶ 继续  |`F5`       |跑到下一个断点或结束 |
|↷ 单步跳过|`F10`      |执行当前行，停到下一行|
|↓ 单步进入|`F11`      |进入函数内部一步步看 |
|↑ 单步跳出|`Shift+F11`|跳出当前函数     |
|⏹ 停止  |`Shift+F5` |结束调试       |

### 调试控制台（DEBUG CONSOLE）

调试停下时，下方会有 **DEBUG CONSOLE** 标签，可以现场敲代码查看变量、做临时计算：

```python
>>> data[0].value    # 查看具体某个值
>>> type(item)       # 看变量类型
>>> total + 100      # 临时计算结果
```

> ⚠️ 注意：在调试控制台里给变量赋值会**真实修改当前调试状态**，比如执行 `total = 999` 会让程序后续真的用 999 继续跑。仅查看用 `print` 类操作不影响，赋值要谨慎。

> **排错思路**：在报错附近打断点 → 启动调试 → 查看变量是不是预期的值 → 找出问题。

## 十、BAT 批处理：将脚本打包成可双击工具

### 什么时候用 BAT

脚本写好调试完毕，进入日常使用阶段后：

- 不想每次都打开 VS Code 再运行
- 需要把工具交给没有 VS Code 的同事使用
- 需要按顺序串联执行多个脚本

> **开发期用 VS Code 调试，使用期双击 BAT 运行。两者不冲突。**

### 创建方式

在项目根目录新建后缀为 `.bat` 的文件，用 VS Code 打开编辑即可。


> ⚠️ **必须用 VS Code 创建和编辑 `.bat` 文件。**
> 如果在桌面用系统自带的"记事本"新建并保存，Windows 记事本可能默认使用 ANSI 编码，这会导致即使加了 `chcp 65001` 依然输出中文乱码。VS Code 右下角状态栏可查看当前文件编码，确认是 UTF-8 即可。

### 场景 1：基础一键运行（uv 项目推荐）

```bat
@echo off
chcp 65001 >nul

echo 开始运行...
uv run main.py
pause
```

每行命令的作用：

|命令               |作用          |不加会怎样          |
|-----------------|------------|---------------|
|`@echo off`      |关闭命令路径回显    |屏幕会刷出一堆内部命令路径，乱|
|`chcp 65001 >nul`|切换为 UTF-8 编码|中文显示成乱码（必加）    |
|`uv run main.py` |自动激活环境并运行   |——             |
|`pause`          |等用户按键再退出    |窗口闪退，看不到结果（必加） |

### 场景 2：串行执行多个脚本（流水线）

```bat
@echo off
chcp 65001 >nul

echo [步骤 1/3] 正在读取检测数据...
uv run step1_read_data.py

echo [步骤 2/3] 正在处理计算...
uv run step2_process.py

echo [步骤 3/3] 正在生成报告...
uv run step3_export_report.py

echo ===== 所有任务执行完毕 =====
pause
```

### 场景 3：传统虚拟环境方式（老项目）

```bat
@echo off
chcp 65001 >nul

echo 正在激活虚拟环境...
call .venv\Scripts\activate.bat

echo 正在运行主程序...
python main.py
pause
```

> ⚠️ **为什么必须用 `call`？**
> 直接写 `.venv\Scripts\activate.bat`（不加 call）相当于把 BAT 控制权**完全转交**给激活脚本。激活脚本执行完就退出，**下面的 `python main.py` 永远不会被执行**。
> 加 `call` 表示”调用一下这个脚本，执行完回来继续往下走”，控制权才能回到主 BAT。

### 完整工程化模板

```bat
@echo off
title 检测报告生成工具
chcp 65001 >nul

echo =========================================
echo         检测报告自动生成系统
echo =========================================

uv run report_generator.py

echo =========================================
pause
```

## 十一、云端防线：GitHub Actions 自动化流水线 (CI)

> - **什么是 CI (持续集成)？**
> - 简单来说，就是你每次向 GitHub 推送（Push）代码时，GitHub 会自动分配一台云端虚拟电脑，帮你把代码拉下来，自动装好 `uv`，自动运行 Ruff 检查规范，自动跑一遍所有的单元测试。
> - **如果全绿**：代码安全，允许合入。
> - **如果爆红**：说明代码有 Bug 或格式不合格，GitHub 会给你发邮件警告，逼你改好再提交。

### 1. 配置云端质检员（YAML 文件）

GitHub 规定，只要你的项目里有 `.github/workflows/` 这个特定文件夹，并在里面放上 `.yml` 配置文件，它就会自动激活 Actions 功能。

在 VS Code 中执行以下操作：
1. 在项目根目录新建文件夹 `.github`（注意前面有个点）。
2. 在 `.github` 里新建文件夹 `workflows`。
3. 在 `workflows` 里新建文件 `ci.yml`。

将以下代码原封不动地粘贴进 `ci.yml` 中：

```yaml
# .github/workflows/ci.yml
name: CI 流水线

# 触发条件：当你向 main 分支 push 代码，或者发起合并请求时执行
on:
  push:
    branches: [ "main" ]
  pull_request:
    branches: [ "main" ]

jobs:
  test-and-lint:
    # 操作系统：GitHub 提供的一台最新的 Windows 虚拟机
    runs-on: windows-latest

    steps:
      # 第 1 步：把你的代码拉取到虚拟机里
      - name: 检出代码
        uses: actions/checkout@v4

      # 第 2 步：极其快速地安装 uv 工具（带缓存机制）
      - name: 安装 uv
        uses: astral-sh/setup-uv@v3
        with:
          enable-cache: true 

      # 第 3 步：让 uv 根据项目配置安装 Python 和所有依赖包
      - name: 还原环境与依赖
        run: uv sync

      # 第 4 步：执行 Ruff 静态检查（格式检查 + 逻辑红线检查）
      - name: 代码规范检查 (Ruff)
        run: |
          uv run ruff format --check .
          uv run ruff check .

      # 第 5 步：执行所有的单元测试
      - name: 运行单元测试 (Pytest)
        run: uv run pytest
```

### 2. 核心原理拆解（为什么这么配？）

* **`uses: astral-sh/setup-uv@v3`**：这是官方提供的现成动作。它不仅帮你装好 `uv`，还会把下载过的依赖**缓存**在 GitHub 服务器上。下次你再 push 代码，这台机器瞬间就能装好环境，不浪费时间。

* **`uv sync`**：这就体现出 `uv.lock` 文件的威力了。云端机器会完全对照你本地锁死的版本号进行安装，**绝对不会**出现“你本地用 pandas 2.1，云端偷偷装了 2.2 导致报错”的灵异事件。

* **`--check` 参数**：注意，我们在云端运行的是 `ruff format --check`。它的意思是“只检查，不帮你改”。既然到了验收环节，质检员只负责挑错（如果格式不对直接让这趟流水线报错），改错是你本地的事。

### 3. ⚠️ 终极防坑：添加兜底测试文件

如果你现在直接 push，流水线在最后一步 `pytest` 时**必定会报错（爆红）**。
为什么？因为 `pytest` 如果在你项目里一个测试用例都没找到，它会认为这是一个错误（Exit code 5）。

为了让第一次流水线顺利跑通，我们需要在本地写一个“兜底”的测试文件。

1. 在项目根目录新建文件夹 `tests/`（与 `src/` 平级）。
2. 在 `tests/` 目录下新建文件 `test_basic.py`。
3. 写入以下测试代码：

```python
# tests/test_basic.py

def test_environment_setup():
    """验证测试环境是否连通"""
    assert True
```

### 4. 触发验收与实战效果

一切就绪后，在 VS Code 终端里执行常规的提交动作：

```bash
git add .
git commit -m "feat: 引入 GitHub Actions CI 流水线"
git push
```

**见证奇迹的时刻：**
1. 马上打开浏览器，进入你的 GitHub 仓库主页。
2. 点击上方的 **"Actions"** 标签页。
3. 你会看到一个正在转圈的黄色图标，点进去。
4. 就像看电影一样，你会看到 GitHub 的服务器正在一行一行地执行你的命令：装依赖、查代码、跑测试。
5. 大约 15 秒后，所有的步骤全部打上 **绿色的勾 (✅)**。

从这一刻起，哪怕是你把代码开放给其他同事协作，只要他们写的代码格式乱七八糟，或者改坏了你原本写好的测试逻辑，这个绿色的勾就会变成刺眼的 **红叉 (❌)**，立刻拦截劣质代码。

>💡使用 `uv run ruff check --fix` 命令，可修复可自动修复的代码

## 十二、分支（Branch）基础

### 什么时候真的需要分支

**日常单人开发，绝大多数时候不需要分支。**直接在 `main` 分支上提交即可，简单直接。

只有以下情况才需要新建分支：

- 想试一个**风险比较高的大改动**，但又怕搞坏现在能跑的版本
- 同时维护**两个并行的版本**（如线上稳定版 + 测试新功能版）
- **多人协作**，每人在自己的分支上开发，最后合并到 main

如果你是单人写检测脚本，十几个项目每个都不大，**99% 的时间一个 main 分支就够**。

### 常用分支操作

```bash
# 查看所有分支（* 号表示当前所在分支）
git branch

# 新建并切换到新分支
git checkout -b feature/新功能名

# 切换回主分支
git checkout main

# 将某分支合并到当前分支
git merge feature/新功能名

# 删除已合并的分支
git branch -d feature/新功能名

# 推送新分支到 GitHub
git push -u origin feature/新功能名
```

### 典型使用场景

```
main 分支         ──────────────────────────────▶ 稳定版本
                              ↑ merge
feature/报告模块  ────────────────▶ 开发完再合并
```

```bash
git remote set-url origin git@github.com:用户名/新仓库名.git
git remote -v   # 验证
```

## 十四、备份策略

> GitHub 不是万能保险箱，重要项目要做多重备份。

**为什么 GitHub 不够？**

- GitHub 服务可能短期不可访问
- 账号可能因误操作被锁
- 仓库可能被误删（Private 仓库删除后短期可恢复，但不要赌）
- 国内网络偶尔抽风访问不上

**推荐的三层备份**

|位置               |用途                         |频率     |
|-----------------|---------------------------|-------|
|**本地 D 盘**       |工作主盘，日常开发                  |实时     |
|**GitHub**       |云端代码备份+版本管理                |每次 push|
|**移动硬盘 / 公司 NAS**|真正的灾备，整个 CodeProjects 文件夹打包|每月 1 次 |

**注意事项**

- ⚠️ **不要把项目文件夹放在 OneDrive、坚果云、百度网盘的同步文件夹里**——这些工具会和 Git 冲突，导致 `.git` 文件夹损坏，整个仓库报废
- ✅ 备份的是**整个项目文件夹**（包含 `.git` 文件夹），这样移动硬盘里的副本本身就是完整 Git 仓库
- ✅ 真正重要的项目，可以同时推到两个 Git 平台（如 GitHub + Gitee 私有仓库）

## 十五、新手常见问题

**Q：报错 `ModuleNotFoundError: No module named 'xxx'`**
→ ① 用 `uv run main.py` 运行而不是 `python main.py`；② VS Code 解释器没选 `.venv`，按 `Ctrl+Shift+P` → `Python: Select Interpreter`。

---

**Q：`git push` 报错 Permission denied**
→ SSH 没配好，执行 `ssh -T git@github.com` 检查。确认用的是 SSH 链接（`git@github.com:...`）而不是 HTTPS（`https://github.com/...`）。

---

**Q：SSH 测试连接没有任何反应或直接失败**
→ 首次连接出现提示时，未输入 `yes` 直接按了回车。重新执行 `ssh -T git@github.com`，看到提示后手动输入 `yes` 再回车。

---

**Q：私钥不小心发给同事 / 上传到 GitHub 了**
→ 立刻按以下步骤处理（顺序不能颠倒）：

1. 在 GitHub 删除对应公钥（Settings → SSH and GPG keys → 点击对应 key 的 Delete）
2. 在本地删除旧密钥：`rm ~/.ssh/id_ed25519 ~/.ssh/id_ed25519.pub`
3. 重新生成新密钥（见第四章第 2 步）
4. 把新公钥添加到 GitHub
5. 如果该密钥还用于其他服务（服务器、Gitee 等），全部解绑重置

> 私钥泄露相当于钥匙丢了，必须立刻换锁。即使没察觉到滥用，也要假设最坏情况。

---

**Q：Git Bash 里 `cd D:\CodeProjects` 报错**
→ Git Bash 用正斜杠，改成 `cd D:/CodeProjects`。

---

**Q：`.venv` 文件夹被上传到 GitHub 了**

```bash
git rm -r --cached .venv/
git commit -m "从仓库移除 .venv"
git push
```

然后确认 `.gitignore` 里有 `.venv/`。

---

**Q：克隆别人的项目后跑不起来**
→ 没执行 `uv sync` 还原依赖。进入项目文件夹后执行 `uv sync`，再用 `uv run main.py` 运行。

---

**Q：代码在我电脑能跑，发给别人报错”找不到文件”**
→ 代码里用了绝对路径（如 `C:\Users\张德帅\...`）。改成相对路径（如 `./data/input.xlsx`）。

---

**Q：BAT 文件双击运行中文显示乱码**
→ BAT 文件开头没加 `chcp 65001 >nul`，加上这行重新运行即可。

---

**Q：BAT 双击后窗口一闪而过**
→ BAT 文件末尾没有 `pause`，加上这行即可。

---

**Q：Windows Defender 报毒，把 BAT 或 Python 脚本删了**
→ 自己写的小工具误报很常见，处理方式：

1. 打开 **Windows 安全中心** → **病毒和威胁防护**
2. 点击 **管理设置** → **添加或删除排除项**
3. 添加 `D:\CodeProjects\` 整个文件夹为排除项

> 仅对自己写的代码这样做，不要排除来路不明的文件。

---

**Q：`git pull` 之后代码被改乱了**
→ 遇到合并冲突，VS Code 文件里会有 `<<<<<<<` 标记，手动选择保留哪部分，保存后重新 add、commit、push。

---

**Q：想撤销还没 push 的最后一次 commit**

```bash
git reset --soft HEAD~1   # 撤销 commit，文件改动保留
```

---

**Q：代码写到一半，需要临时去改另一个问题**

```bash
git stash        # 储藏当前改动
# 去处理其他问题 ...
git stash pop    # 回来后恢复
```
