下面这份是我从大量项目里看到的、跨语言跨领域都成立的工程习惯。按"实施成本 vs 长期收益"排序，前几条是性价比最高的，建议无脑执行。

---

## 一、代码组织（最重要）

1. **业务逻辑、IO、UI 三层分离**
   - 业务函数纯参数纯返回，不打印、不读盘、不弹窗
   - IO 层（文件、数据库、网络、subprocess）单独成模块，给业务层注入
   - UI 层只在程序入口或 GUI 文件里
   - **判定标准**：你的核心函数能不能在没有真实文件的环境下跑测试？能 = 分离到位

2. **公共能力沉淀，第二次复制就要抽**
   - 第一次写：原地写
   - 第二次写：留意，复制了
   - 第三次写：**必须**抽到 `common/` / `utils/` / `lib/`，不能再复制
   - 抽出来的函数纯参数化，不要带任何"上下文假设"

3. **配置外置，不要全局可变状态**
   - 不要用 `class Config: SOME_PATH = "..."` 这种全局类属性当配置传递
   - 改成函数参数 / dataclass / 配置文件（YAML/JSON/TOML）+ 启动时一次加载传入
   - 全局可变态会让函数突然依赖"调用顺序"，bug 极难定位

4. **一个文件一件事，命名是动词**
   - 工具脚本：`sort_photos.py` / `export_images.py` / `check_links.py`
   - 库模块：`word_helpers.py` / `excel_helpers.py`（名词，描述能力域）
   - 反例：`utils.py`、`helpers.py`、`tools.py`、`misc.py` —— 这种名字一旦出现，3 个月后里面会塞满不相干的代码

5. **目录结构反映"读者关心什么"**
   - 应用类项目：`src/` 或 `项目名/` + `tests/` + `docs/` + `scripts/`
   - 工具集类项目：动词文件平铺 + `common/` 共享
   - 库类项目：`项目名/__init__.py` + 子模块
   - 不要套娃太深，三层目录基本是上限

## 二、函数与接口设计

6. **函数小，单一职责**
   - 一个函数 ≤ 50 行是经验阈值，超过就考虑拆
   - 一个函数做"一件能用动词描述的事"
   - 反例：`process_data()` —— 处理什么？怎么处理？应改成 `validate_then_normalize_then_save_user_records()`，名字太长说明本来就该拆 3 个

7. **数据契约用 dataclass / TypedDict / Pydantic**
   - 跨函数传递的字典就是定时炸弹，字段改名所有调用方默默坏掉
   - Python：标准库 `@dataclass`（足够用）/ Pydantic（要校验时）
   - TS：`interface` / `type`
   - Go：`struct`
   - 总之**不要让结构化数据靠"约定"在多文件流转**

8. **失败用异常，不要用魔法返回值**
   - 不要 `return None`、`return -1`、`return ""` 来表示失败
   - 抛具体异常类：`FileNotFoundError`、`ValidationError`、自定义 `MyDomainError`
   - 调用方在外层 `try` 里能精确捕获想处理的那一类，其他的让它崩

9. **默认参数不要用可变对象**
   - `def f(items=[])` 是 Python 经典坑 —— 多次调用共享同一个 list
   - 用 `def f(items=None): items = items or []`

10. **早 return，不要深嵌套**
    - 处理边界条件先 return，主流程左对齐
    - 4 层缩进基本是上限，再深就抽函数

## 三、错误处理

11. **边界严格，内部信任**
    - 在系统边界（用户输入、外部 API、文件读取）做严格校验
    - 内部模块之间互相信任 —— 不要每个函数都重复检查 `if x is None`
    - 内部错误就让它崩，因为崩本身就是一个清晰信号

12. **不要吞异常**
    - `except Exception: pass` 是反人类，必查必删
    - 至少要 `logger.exception(...)` 打出来
    - 要重试 / 要兜底就明确写出来，并在日志里说明为什么这里要兜底

13. **资源用 `with` / `try-finally` / RAII**
    - 文件、连接、锁、Word COM、临时目录 —— 一律用上下文管理器
    - 不依赖"我记得最后会关掉" —— 异常分支会绕过
    - Python：`with open(...)`、`contextlib.contextmanager`
    - 自定义资源就实现 `__enter__/__exit__`

14. **错误信息含上下文**
    - 不要 `raise ValueError("invalid")` —— invalid 啥？
    - 要 `raise ValueError(f"列名 '{col}' 在 sheet '{sheet}' 中不存在，可用列: {df.columns.tolist()}")`
    - 看日志的人不知道现场状态，你必须把现场信息塞进消息里

## 四、可观测性

15. **日志分级 + 结构化**
    - DEBUG/INFO/WARN/ERROR 分清楚
    - 关键流程节点必须有 INFO 日志（"开始处理 X"、"完成 X，耗时 Y"）
    - 生产项目用 `logging` + JSON 格式化，本地脚本可以 print + emoji（你 02_Core 就是这风格）
    - **重要**：长任务（>3 秒）必须打进度，不然用户永远不知道是卡了还是在跑

16. **stdout 行缓冲 + flush=True**
    - Python 默认块缓冲，IDE / 重定向场景会让 `print` 攒一堆才输出
    - 启动时 `sys.stdout.reconfigure(line_buffering=True)` 或 `python -u` 运行
    - 这条吃了几次亏后会刻进肌肉记忆

17. **错误堆栈完整保留**
    - `except E as e: raise OtherError(...) from e` —— 用 `from` 保留原始 traceback
    - 不要 `except: raise Exception(str(e))` —— 这会把堆栈截断，调试地狱

## 五、可测试性

18. **依赖注入，不要硬编码外部依赖**
    - 函数里别 `requests.get(URL)` 直接发请求 —— 测试没法 mock
    - 改成 `def f(http_client): http_client.get(...)`，测试时传假的
    - 时间、随机数、文件系统、网络、当前目录 —— 全是可注入候选

19. **一个文件一个测试文件**
    - `foo.py` 对应 `tests/test_foo.py`
    - 测试金字塔：单元测试多、集成测试中等、端到端少
    - 测试是工具，不是任务 —— 写测试比 print 调试更快

20. **副作用容器化 / 沙箱化**
    - 写文件去 `tmp_path`（pytest fixture）
    - 改环境变量用 `monkeypatch`
    - 测完自动清理，不污染开发环境

## 六、版本控制

21. **小步提交，每个 commit 是一个原子改动**
    - 一个 commit 只做一件事：要么修 bug，要么加功能，要么重命名，**不要混**
    - 重构和功能改动分开提，code review 时一眼看出意图

22. **commit 信息写"为什么"，不写"做了什么"**
    - 反例：`update foo.py`
    - 好例：`fix: 用 DispatchEx 替代 Dispatch 防止 Word COM 附着到僵尸进程`
    - diff 已经告诉读者"做了什么"，commit 信息要补充"为什么这样做"

23. **`.gitignore` 一开始就写好**
    - `__pycache__/`、`.venv/`、`*.log`、`.env`、IDE 配置、临时文件、构建产物
    - 一旦提了进去再删，历史里永远有

24. **绝不提交 secrets**
    - `.env` 不进库，`.env.example` 进库
    - API key / 密码 / 证书走环境变量或密钥管理服务
    - 提了进去就当作泄露了，立刻 rotate，别想着 `git rm` 就完事

25. **分支命名规范**
    - `feat/xxx`、`fix/xxx`、`refactor/xxx`、`docs/xxx`
    - 多人协作时用 `username/feat-xxx`

## 七、环境与依赖

26. **每个项目独立虚拟环境**
    - Python：venv / uv / poetry / conda 选一个，不要混
    - Node：每个项目自己的 `node_modules`
    - 不要全局 pip install，全局污染卸载都难

27. **lock 文件入库**
    - `uv.lock` / `poetry.lock` / `package-lock.json` / `Cargo.lock` 必须提交
    - 保证别人 clone 下来装出来的依赖版本和你完全一致

28. **依赖最小化原则**
    - 加每个新依赖前问：能不能用标准库？能不能 20 行实现？
    - 依赖每多一个，就多一个安全更新源、版本冲突源、维护负担
    - 反例：为了"美化字符串"装一个 1MB 的库

29. **区分运行依赖 vs 开发依赖**
    - 运行：`requirements.txt` / `pyproject.toml [project.dependencies]`
    - 开发：`requirements-dev.txt` / `[project.optional-dependencies.dev]`
    - 部署时只装运行依赖，少装一堆 pytest/black/mypy

## 八、性能与资源

30. **先测量再优化**
    - 不要凭直觉改 —— 用 profiler 找出真正的瓶颈
    - Python：`cProfile` / `py-spy`；前端：Chrome DevTools；JVM：JFR
    - 80% 的"性能优化"是改了不影响瓶颈的地方

31. **批量优于循环单条**
    - 数据库：`INSERT ... VALUES (...), (...), (...)` 一次插 1000 条 vs 循环 1000 次
    - HTTP：keep-alive / 连接池 / 并发请求
    - 文件：一次 `read()` 全读 vs 多次小 read
    - 一次 IO 的固定开销 >> 数据传输开销

32. **大文件 / 大数据流式处理**
    - 不要 `data = f.read()` 把整个 10G 文件读进内存
    - 用 generator / iterator / chunked read
    - SQL 用 server-side cursor，pandas 用 chunksize

33. **缓存有 TTL 和容量上限**
    - `functools.lru_cache(maxsize=128)`，别 `maxsize=None` 任其膨胀
    - 时间敏感数据加 TTL，不然永远拿到旧值

## 九、安全（即使是内部工具也要养成习惯）

34. **永远不要拼接 Shell / SQL**
    - SQL：参数化查询 `cursor.execute("... WHERE id = ?", (id,))`
    - Shell：用列表形式 `subprocess.run(["cmd", arg1, arg2])`，**不要** `shell=True` + 字符串
    - 命令注入 / SQL 注入是 OWASP Top 10 常驻成员

35. **路径要规范化、要校验**
    - 用户传的路径用 `os.path.realpath()` 解析
    - 校验是否在期望根目录内：防"路径遍历"（`../../../etc/passwd`）
    - Windows：注意大小写不敏感、保留名（`CON`, `PRN`, `AUX`）

36. **不要 log 敏感信息**
    - 密码、token、信用卡号、身份证 —— 日志里要脱敏
    - 异常对象 `str(e)` 时检查不要带敏感数据

37. **第三方输入永远不可信**
    - HTTP body / query / header / cookie / 文件名 / 文件内容 —— 全是攻击面
    - JSON 反序列化要限制大小和深度
    - 反序列化用户内容绝不用 `pickle` / `eval` / `yaml.load`（用 `safe_load`）

## 十、文档与命名

38. **README 必备四块**
    - 这是什么（一句话）
    - 解决什么问题（场景）
    - 怎么跑起来（5 分钟内能跑通）
    - 怎么贡献 / 怎么找帮助
    - 缺这四样的项目，6 个月后你自己都看不懂

39. **docstring 写"为什么"，不写"做什么"**
    - 反例：`# 把 a 加上 b`（看代码就知道）
    - 好例：`# 用 a + b 而不是 a | b 是因为 b 可能为负，需要保留符号`
    - 注释解释意图、限制、坑、为什么这样而不是那样

40. **命名规则**
    - 函数：动词开头 `parse_xxx` / `validate_xxx` / `compute_xxx`
    - 类：名词 `UserSession` / `RequestHandler`
    - 布尔：`is_xxx` / `has_xxx` / `should_xxx` / `can_xxx`
    - 常量：`SCREAMING_SNAKE_CASE`
    - 私有：`_leading_underscore`

41. **不要写"过期"注释**
    - 删除的代码就删干净，不要 `// removed: blah blah`
    - "TODO" 必须带姓名 + 日期 + 链接（`# TODO(zgq, 2026-04, #issue-123)`），无主 TODO 等于垃圾

## 十一、跨平台 / 可移植性

42. **路径用 `pathlib.Path` 或 `os.path.join`**
    - **不要**写死 `\\` 或 `/`
    - Windows 路径有 `\` `/` `:` 大小写不敏感等坑
    - 用 `Path("a") / "b" / "c"` —— 跨平台、可读

43. **编码全程 UTF-8**
    - 文件读写显式 `encoding="utf-8"`
    - 不要靠平台默认（Windows 是 GBK，Linux 是 UTF-8，会爆中文乱码）
    - 命令行子进程也要 `text=True, encoding="utf-8"`

44. **换行符 / Shell 差异**
    - Windows `\r\n` vs Unix `\n` —— git 用 `core.autocrlf` 配置
    - Shell 命令：bash 不一定有，不要假设 `grep`/`sed` 在 Windows 也在
    - 用 Python / 跨平台 CLI 工具（如 `ripgrep`、`fd`）替代

45. **时区**
    - 内部时间一律 UTC，展示层再转本地时区
    - `datetime.now()` 是 naive，是 bug 之源 —— 用 `datetime.now(tz=timezone.utc)`

## 十二、AI 协作时代的新习惯

46. **类型注解全开**
    - 不只是给 IDE 用，也是给 AI 看的
    - AI 看到 `def parse(s: str) -> Photo` 比看到 `def parse(s)` 能给出准确得多的补全和重构
    - Python：`from __future__ import annotations` + mypy/pyright

47. **函数小 + 命名清楚 = AI 友好**
    - AI 上下文有限，巨型函数它读不全
    - 函数命名清晰意味着 AI 不需要看实现就能 "callsite" 推理
    - 这条和给人读的好习惯**完全一致**，AI 时代只是奖励变大了

48. **测试先行 = AI 自动验收**
    - 让 AI 写实现时，先把测试给它，它能自己迭代到通过
    - 没有测试，AI 给你的代码是"看起来对"

49. **README + AGENTS.md / CLAUDE.md 写明项目约定**
    - 让 AI 一进来就知道：技术栈、目录结构、命名规则、提交规范、运行命令
    - 比每次对话重新解释要省 10 倍时间

50. **保留人审，特别是高风险操作**
    - 删除、迁移、推送、改权限、改 CI、改生产配置 —— 即使 AI 写好了也手动跑
    - AI 工具用得越深，"人工守门"越关键

## 高杠杆精选（如果只能记 10 条）

如果上面 50 条记不住，至少把这 10 条做到，已经能甩开 80% 的项目：

1. **业务/IO/UI 分离** —— 函数能在没有真实环境下被调用 = 万事可期
2. **复制第二次就抽** —— 不抽不让自己写第三遍
3. **数据契约用 dataclass** —— 别让 dict 满天飞
4. **失败用异常 + 错误带上下文** —— `raise ValueError(f"...具体值...")`
5. **资源用 `with`** —— 永远别裸 open
6. **小步 commit + 写为什么** —— 6 个月后的自己感谢你
7. **每个项目独立 venv + lock 文件入库** —— 环境可复现
8. **不要凭直觉优化，先 profile** —— 80/20 法则
9. **路径 pathlib + 编码 UTF-8 显式** —— 跨平台不踩坑
10. **类型注解全开 + 函数小** —— 人和 AI 都受益
