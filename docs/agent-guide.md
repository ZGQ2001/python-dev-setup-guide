# Agent 模式详解：让 AI 在你的项目里自己动手干活

> 💡 本章是 2.1 章"Vibe Coding"的进阶内容。建议你已经按 2.1 章的方法用网页版 AI 跑了 2-3 个小项目后再来看本章。Agent 比网页版 AI 更强大，但踩坑也更疼——先走稳，再跑。

---

## 〇、前置准备：AGENTS.md —— 让 AI 读懂你的项目

在让 Agent 动手之前，你必须先告诉它"你是谁、用什么工具、文件放哪"。这跟 2.1 章第五节说的 `项目说明.txt` 是同一个道理，但更正式——Agent 打开项目时**自动读取**，你不用每次粘贴。

### AGENTS.md 是什么

`AGENTS.md` 是一个放在项目根目录的 Markdown 文件，内容是写给 AI 看的——"这个项目干嘛的、用了什么工具、有什么规矩"。

### 和 `项目说明.txt` 的区别

| 方式 | 用法 | 适合谁 |
|------|------|--------|
| `项目说明.txt` | 每次对话手动复制粘贴 | 只用**网页版 AI**（豆包网页、DeepSeek 网页），不在编程软件里用 AI 插件 |
| `AGENTS.md` | 放项目根目录，AI **自动读取** | 用 Copilot / Cursor / DeepSeek TUI 直接在项目里工作 |

### 最简模板

在你的项目根目录（和 `pyproject.toml` 同一层）新建一个 `AGENTS.md` 文件，复制以下内容，改成你自己的项目信息：

```markdown
# 项目说明

- **项目名**：工资管理
- **语言/版本**：Python 3.12
- **包管理器**：uv
- **代码检查**：Ruff
- **数据文件夹**：data/
- **输出文件夹**：output/
- **源代码**：src/工资管理/

## 运行方式
uv run python src/工资管理/main.py

## 注意事项
- 所有路径用正斜杠 / 不要用反斜杠 \
- 不要手动修改 pyproject.toml，用 uv add 添加依赖
- 不要手动改 uv.lock
- 数据文件放 data/，输出文件放 output/
```

### 放在哪里

```
你的项目名/
├── AGENTS.md          ← 放这里（和 pyproject.toml 同级）
├── pyproject.toml
├── src/
│   └── 工资管理/
│       └── main.py
├── data/
├── output/
└── ...
```

保存以后，每次用 Agent 打开这个项目，它自动就知道环境、路径和规矩。不用每次粘贴。

> 💡 不同工具有不同叫法：Cursor 用 `.cursorrules`，GitHub Copilot 用 `.github/copilot-instructions.md`。但 **`AGENTS.md` 是当前最通用的格式**——DeepSeek TUI、Claude Code、Cline 等都认它。先从 AGENTS.md 开始，将来需要时再按特定工具补充。

## 一、先搞清楚：Agent 到底是什么？

在 2.1 章你用的是**对话模式**：

```
你说话 → AI 回一段代码 → 你复制 → 粘贴到 VS Code → 手动跑 → 出错了再贴回去
```

Agent 模式把这个流程**反过来**：

```
你说话 → AI 自己打开文件 → 自己写代码 → 自己跑 → 看到报错自己改 → 再跑 → 直到通过
```

你不是在和 AI 聊天。你是给 AI **分配了一个任务**，然后看它干活。

> **一句话**：网页版 AI 是你问一句它答一句的"顾问"；Agent 是你交代完就去喝咖啡、回来验收的"实习生"。

---

## 二、Agent 能干什么？

以小王为例。小王现在用网页版 AI 写出工资计算器，每次考勤表更新后手动 `uv run`。但如果他想让 AI 帮他做这些：

| 任务 | 网页版 AI | Agent 模式 |
|------|---------|-----------|
| 写一个功能 | ✅ 可以 | ✅ 可以 |
| 写完自动跑测试看对不对 | ❌ 你要自己跑 | ✅ Agent 自己跑 |
| 跑不过自己改 | ❌ 你要贴报错 | ✅ Agent 读到报错自己修 |
| 改了代码把别的功能弄坏了 | ❌ 你自己发现 | ✅ Agent 会跑全部测试 |
| 同时改 3 个文件协调工作 | ❌ 你要手动分3次 | ✅ Agent 一次全改 |
| 根据你的项目目录结构自动找文件 | ❌ 你要告诉它路径 | ✅ Agent 自己扫描 |

## 三、市面上的 Agent 工具：三款最主流

### 1. GitHub Copilot Agent（推荐新手首次尝试）

Copilot Agent 是 VS Code 的一个模式，你在聊天框里打字，它在你的项目里干活。

- **怎么打开**：VS Code 装好 Copilot 插件 → 按下 `Ctrl+Shift+I` 打开 Copilot Chat → 模式选 **Agent**（不是 Ask，不是 Edit）
- **免费额度**：每月 50 次 Agent 对话（2026 年标准，够入门练手）
- **适合谁**：已经配好第一章 VS Code 环境的用户
- **最大优点**：和 VS Code 无缝集成，改了什么你能在编辑器中看到 diff（差异对比），随时可以撤销

### 2. Cursor

Cursor 是一个独立的编程软件（不是 VS Code 插件），Agent 内建。

- **下载**：cursor.com
- **免费额度**：每月 2000 次补全 + 有限 Agent 次数
- **适合谁**：不想折腾 VS Code 配置、想开箱即用的用户
- **最大优点**：对项目上下文的理解比 Copilot 更深，Agent 行为更精准

### 3. DeepSeek TUI / Claude Code（不推荐新手）

这类是纯命令行的 Agent，没有图形界面，在终端里打字。功能最强但门槛最高。

- **为什么先不碰**：你需要习惯终端操作、理解 Git 分支、知道怎么恢复被 Agent 改坏的文件
- **什么时侯再学**：Copilot Agent 用满 50 次、已经熟练了"分配任务 → 验收 → 撤销"整个循环

> 💡 **推荐路径**：Copilot Agent（免费、有撤销保护） → 用熟了 → Cursor（更深度的项目理解） → 再考虑命令行 Agent。

## 四、实操：跟小王学用 Copilot Agent

小王已经用网页版 AI 写出了工资计算器（`src/工资管理/main.py`）。现在他想让 Agent 帮他加一个新功能：**按部门统计平均工资，生成一张汇总表。**

### 第 1 步：切换模式

在 VS Code 中，`Ctrl+Shift+I` 打开 Copilot Chat。在输入框左下角，确保选中的是 **Agent** 模式（不是 Ask）。

### 第 2 步：描述任务

在输入框里打：

```
帮我在这个项目里加一个功能：

1. 读取 output/工资条.xlsx
2. 按"部门"列分组，计算每个部门的平均实发工资
3. 结果保存为 output/部门汇总.xlsx

源文件在 src/工资管理/main.py。
```

按回车。

### 第 3 步：看 Agent 干活

Copilot Agent 会开始自动操作——你会在 VS Code 里看到：

1. Agent 读了一遍你的 `main.py`，理解了当前代码
2. 打开终端，运行了一次 `uv run python src/工资管理/main.py` 确认现有功能正常
3. 在 `main.py` 里追加了新的函数
4. 又跑了一遍，确认新功能没把旧功能弄坏
5. 终端显示：`✅ 部门汇总已生成：output/部门汇总.xlsx`

整个过程 30-60 秒。你没有写任何代码。

### 第 4 步：验收（这一步不能省！）

Agent 干完活后，你不要直接关掉 VS Code。做三件事：

1. **看文件变更**：在 VS Code 左侧 Source Control 面板里，看 Agent 改了哪些文件、改了什么行。如果改了你不想改的地方，右键 → 撤销。

2. **跑一遍结果**：打开 `output/部门汇总.xlsx`，目测：
   - 部门数量对吗？（你的公司有 5 个部门，汇总表应该有 5 行）
   - 平均工资合理吗？（技术部平均 8000 左右，不可能 200 或 20 万）

3. **跑旧功能**：重新跑一次工资计算，确认 Agent 没有把原来的计算逻辑改坏。

## 五、⚠️ Agent 模式的三大坑（每个都有人踩过）

### 坑 1：Agent 会改你不该改的东西

Agent 为了"完成任务"，可能会修改你项目的配置文件（`pyproject.toml`）、自动装你没想装的库、甚至改你的 `.gitignore`。

**防护**：
- 每次 Agent 干完活，打开 VS Code 的 Source Control 面板，**逐文件检查变更**。
- 如果你不想让它动某个文件，在对话里说："不要修改 pyproject.toml，不要安装新库。"
- Git 是你的救命稻草：**Agent 干活前先 commit 一次**，干完不满意直接 `git checkout .` 全部撤销。

### 坑 2：Agent 在不报错的情况下输出错误结果

这个坑和网页版 AI 一样，但 Agent 更危险——因为它自动跑、自动改、自动验证，你容易不知不觉就接受了错误结果。

**防护**：
- 你必须目测输出。Agent 说"通过测试"不等于结果正确——测试也可能是 Agent 自己写的、恰好通过了而已。
- 回想 2.1 章引子部分说的：**你是最后的质量底线。**这个原则在 Agent 模式下翻倍重要。

### 坑 3：Agent 陷入"改-错-改-错"的死循环

Agent 有时会"修了一个 bug 但引入三个新 bug"，然后不断修、不断错，直到把代码搅成一锅粥。更糟的是，它可能在一个循环里烧掉你大量 API 额度。

**防护**：
- 如果 Agent 在同一件事上改了 3 次还没对，**立即停止**。用 `Ctrl+C` 中断。
- 切回 Ask 模式（不是 Agent），把当前的问题用文字描述给 AI，让它给一个"静态方案"——也就是回到网页版 AI 的用法：它给你代码，你手动改。
- 等静态方案跑通后再切回 Agent。

## 六、Agent 模式下的 Git 铁律

用 Agent 模式之前，你必须已经掌握了 1.4 章的 `git add → commit → push`。原因很简单：

```
Agent 干活前：git commit -m "Agent 改之前的状态"
Agent 干完活：验收
  → 满意：git commit -m "Agent 帮忙加了部门汇总功能"
  → 不满意：git checkout .   （一秒回到改之前，不留痕迹）
```

> ⚠️ **不用 Git 就用 Agent，等于不开降落伞跳伞。** Git 是你唯一能"后悔"的机制。

## 七、什么时候该用 Agent，什么时候不该用

| 场景 | 用 Agent？ | 原因 |
|------|-----------|------|
| 在一个全新的空项目里从零搭建 | ✅ 可以 | Agent 不会弄坏任何现有代码 |
| 给现有项目加一个独立的新功能 | ✅ 可以 | 影响范围可控，容易验收 |
| 改一个涉及 3 个以上文件的 bug | ⚠️ 谨慎 | Agent 可能改错依赖关系，验收难度大 |
| 改核心计算公式（如工资算法、财务逻辑） | ❌ 不建议 | 这种逻辑你比 Agent 懂，让网页版 AI 给你方案，你手动改 |
| 整理代码风格、加注释 | ✅ 很适合 | 纯体力活，Agent 最擅长 |
| 在不懂 Git 的情况下 | ❌ 绝对不能 | 先去学 1.4 章 |

---

## 🏆 本章小结

- Agent 是"你交代任务、AI 自己动手"的工作模式——你把重复劳动外包给 AI，你的角色从"写代码的人"升级为"验收的人"。
- 你的责任不但没减少，反而加重了：Agent 能干更多活，意味着它能犯更多错。你必须目测输出、核对数量、守住业务边界。
- Git 是 Agent 模式的必备保险——每次让 Agent 干活前先 commit，不满意就撤销。
- 新手从小任务、新功能开始，不要在不懂 Git 的情况下让 Agent 动核心代码。
