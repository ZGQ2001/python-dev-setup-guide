# AI 辅助编程工作流

> 适用读者：已经完成第一章环境配置，开始正式写代码的新手。
> 前置要求：Copilot 插件已安装（见 1.2），基本了解 VS Code 操作。

---

## 一、重新定位：你和 AI 各自做什么

### AI 是"代码草稿机"，你是工程决策人

新手最容易踩的坑是把 AI 当**正确答案生成器**——贴一段需求，拿回来直接用，报错也不知道为什么。

正确的心态是把 AI 当成一个**打字极快、知识面广、但从不为自己代码负责的实习生**：

```
你（工程决策人）                  AI（代码草稿机）
───────────────                  ──────────────
决定"做什么"                     生成"怎么做"的初稿
判断方案是否合理                  按你的指令机械执行
验证正确性                        不验证自己的输出
对最终结果负责                    不承担任何责任
```

> 💡 **核心铁律：AI 写代码，你签验收单。** 就像施工现场——分包队干了活，监理不验收不能报验。AI 输出的每一行代码，没有经过你的三关验收（见第四节），都只是草稿。

### 责任边界对照表

| 环节 | AI 负责 | 你负责 |
|------|---------|--------|
| 理解业务需求 | ❌ | ✅ 只有你知道"锚杆拉拔力超过 120kN 才算异常" |
| 分解任务 | ⚠️ 可以帮忙拆，但你要确认 | ✅ 最终拍板 |
| 写代码初稿 | ✅ 这是它最擅长的 | ❌ |
| 选第三方库 | ⚠️ 会推荐，会编造不存在的库 | ✅ 核实库是否真实存在、版本是否匹配 |
| 处理敏感数据 | ❌ 绝对不能把 API 密钥、客户数据贴给 AI | ✅ 用 `.env` 隔离（见 2.4） |
| 跑通代码 | ⚠️ 能帮你调试 | ✅ 最终必须在你电脑上 F5 跑通 |
| 代码风格统一 | ✅ Ruff 格式化 | ✅ 配置一次 Ruff 规则（见 3.1） |
| 提交到 GitHub | ❌ | ✅ 你按提交键，你写 commit message |
| 出事故背锅 | ❌ AI 没有任何法律/职业责任 | ✅ 代码上挂着你的名字 |

> ⚠️ **红线**：涉及客户隐私、检测原始数据、公司内部系统的账号密码——**绝对不要粘贴到 AI 对话框里**。ChatGPT / Claude / Copilot Chat 的对话会被服务商用于训练（除非你购买了企业版数据隔离）。

## 二、工具矩阵：Copilot vs Claude/ChatGPT

两种工具不是"谁更好"的关系——它们干的事不同。

### 场景分工表

```
你是写代码的工人，面前有两样东西：

┌─ Copilot（在你右手边）─────────────────┐
│  像影子一样跟着你的手指                  │
│  你打字，它猜你接下来要写什么             │
│  不需要切窗口，不需要打字提问             │
│  适合：写重复代码、填函数体、写测试      │
└────────────────────────────────────────┘

┌─ Claude / ChatGPT（在另一个屏幕上）──────┐
│  像一个坐在旁边的老工程师                 │
│  你描述要做什么，它写一整段出来            │
│  适合：从零生成文件、解释报错、重构      │
└────────────────────────────────────────┘
```

| 场景 | 用 Copilot | 用 Claude / ChatGPT |
|------|-----------|-------------------|
| 写一个 for 循环读 Excel 所有 Sheet | ✅ 手写到一半，Tab 补全 | ❌ 杀鸡用牛刀 |
| 从零设计一个数据库写入模块 | ❌ 太大，补全不了 | ✅ 描述需求，生成完整类 |
| 报错信息看不懂 | ⚠️ Copilot Chat 可用 | ✅ 贴报错 + "什么意思？怎么修？" |
| 已有代码需要重构 | ❌ | ✅ 贴代码 + "改成 dataclass，去掉冗余" |
| 写 20 个相似的测试用例 | ✅ 写完第 1 个，后面自动补 | ⚠️ 可以但要写长提示 |
| 学习一个没见过的库 | ❌ | ✅ "pandoc 怎么把 docx 转 markdown？给个最小例子" |

### Copilot 常用快捷键

| 快捷键 | 功能 | 什么时候用 |
|--------|------|-----------|
| `Tab` | 接受灰色建议 | Copilot 猜对了，一拍即合 |
| `Esc` | 拒绝建议 | 猜错了，当没看见 |
| `Alt + ]` | 看下一条建议 | 第一条不满意，翻翻备选 |
| `Alt + [` | 看上一条建议 | 翻过头了，往回找 |
| `Ctrl + Enter` | 打开 Copilot 建议面板 | 同时看多条建议，挑最好的一条 |
| `Ctrl + I` | 打开 Copilot Chat 内联对话 | 选中一段代码 → 直接对话改它 |
| `Ctrl + Shift + I` | 打开 Copilot Chat 侧边栏 | 完整对话窗口 |

> 💡 新手最容易忽略的快捷键是 **`Ctrl + I`**（内联对话）。选中任意代码 → `Ctrl + I` → 输入"给这个函数加类型注解" → Copilot 在原位置直接改好。不需要复制粘贴到外部对话。

## 三、有效提示词：让 AI 准确理解需求

### 三要素：背景 + 输入格式 + 期望输出

AI 不会读心。你脑子里知道的上下文，AI 一概不知。提示词的质量直接决定代码质量。

**坏提示 vs 好提示对比（用检测数据场景）：**

| 场景 | ❌ 坏提示 | ✅ 好提示 |
|------|----------|----------|
| 数据读取 | "帮我读 Excel" | "我有一个 Excel 文件 `./data/锚杆拉拔.xlsx`，第一行是表头，列名包括：锚杆编号、试验荷载(kN)、位移(mm)。帮我用 openpyxl 写一个函数，返回 `list[dict]`，每行是一个锚杆的数据。" |
| 数据处理 | "把这些数据画个图" | "用 matplotlib 画锚杆荷载-位移曲线。X 轴是位移(mm)，Y 轴是荷载(kN)。每个锚杆一条线，图例显示编号。DPI 300，保存到 `./output/曲线图.png`" |
| 格式转换 | "转一下格式" | "把 `data.json` 里的检测记录转成 docx 报告。模板用 `./templates/report.docx`，需要替换的占位符是 `{{检测日期}}`、`{{构件编号}}`、`{{检测结论}}`" |
| 代码理解 | "这段代码什么意思" | "这是同事留下的 `data_cleaner.py`，我标注了三个看不懂的地方（`# ← 这里`），帮我解释每个标注位置在干什么，以及有没有更简单的写法。" |

### 好提示的通用公式

```
【背景】我正在做什么项目，用了什么库，数据长什么样
【输入】我给你什么（文件路径 / 代码片段 / 报错信息）
【输出】我要你返回什么（代码 / 解释 / 修改建议），格式要求（函数签名 / 返回值类型）
```

### 迭代追问模板

AI 第一次给的答案很少一次到位。像跟实习生说话一样，不满意就追。

```
第一轮：
  "帮我写一个函数，读取锚杆检测 Excel，返回 list[dict]"

AI 生成后你发现没处理缺失值：

第二轮：
  "这个版本不错。但 Excel 里有空单元格，空值应该存为 None 而不是 ''。
   另外加个验证：荷载值为负的时候抛出 ValueError。"

AI 改完后你觉得性能不够：

第三轮：
  "如果 Excel 有 50000 行，现在的逐行读取会很慢。
   能不能用 openpyxl 的 read_only 模式？给我看改后的版本。"
```

> ⚠️ 不要一次提三个需求——AI 会顾此失彼。**一需求一追问**，每次只改一个维度。

## 四、验收 AI 代码：三关必过

**AI 生成的代码，在通过下面三关之前，只是"草稿"。严禁直接提交。**

### 第一关：读懂

逐行看一遍，确认你理解每一行在干什么。

| 排查点 | 看什么 | 为什么重要 |
|--------|--------|-----------|
| 硬编码路径 | `C:\\Users\\xxx\\data.xlsx` | 换电脑就挂。必须改成相对路径 `./data/data.xlsx` |
| 陌生 import | `from cryptography import ...` | AI 有时引入你不需要/不存在的库。确认：①这个库真实存在吗？②项目真需要它吗？ |
| 魔法数字 | `if value > 150:` | 150 是什么？荷载阈值？哪个规范定的？必须改成命名常量 `MAX_LOAD_KN = 150` |
| 假函数调用 | `df.to_civ_format()` | AI 会编造不存在的 API。必须确认这个函数/方法真实存在。 |

**实操：** 选中 AI 生成的代码 → `Ctrl + I` → 输入"逐行解释这段代码在做什么"。Copilot 会用中文解释逻辑。你能看懂解释 → 过关。

### 第二关：红线全清

VS Code 底部状态栏，**Pylance 和 Ruff 必须零报错**。

```
Pylance（静态检查）：
  ❌ 1 error, 3 warnings    ← 第一关没过，不许往下走
  ✅ 0 errors, 0 warnings   ← 可以进第三关

Ruff（格式 + 代码质量）：
  ❌ 文件旁边有红色波浪线    ← Ctrl+S 保存会自动修一部分
  ✅ 无波浪线               ← 过关
```

**为什么这一关是红线？**
Pylance 能在代码跑之前就发现类型错误（比如把字符串当数字做加法）。很多 AI 生成的代码类型混乱——Pylance 是第一道防线。如果 Pylance 报错你没清就提交，等于带着已知隐患上路。

### 第三关：F5 跑通主路径

不只是"不报错"——要验证输入输出正确。

**实操流程：**

```bash
# 1. 在终端跑一次
uv run python src/你的模块.py

# 2. 如果函数需要参数，写一个简单的测试调用
# 在文件末尾加：
if __name__ == "__main__":
    result = 你的函数("./data/test.xlsx")
    print(f"返回了 {len(result)} 条记录")  # 目测数量对不对
    print(result[0])                         # 目测第一条数据结构对不对

# 3. 故意给错误输入，看报错信息是否清晰
result = 你的函数("./data/不存在的文件.xlsx")  # 应该报 FileNotFoundError，不是奇怪的报错
```

> ⚠️ **AI 代码最常见的假象**：不报错但输出为空、或者静默跳过一半数据。必须用真实数据目测输出，不能只看"没报错"就过关。

## 五、让 AI 持续输出高质量代码的习惯

### 习惯 1：始终要求类型注解

每次让 AI 写函数，明确要求返回值类型：

```
✅ "写一个函数 read_bridge_data(path: str) -> list[dict]，返回每条检测记录"
```

**为什么类型注解这么重要？**
Pylance 靠类型注解做静态验证。你写了 `-> list[dict]`，如果函数内部某处返回了 `None`，Pylance 会立刻标红——代码还没跑，bug 已经被拦住了。没有类型注解的函数，Pylance 只能假装看不见。

### 习惯 2：要求小函数，拒绝大块代码

AI 特别喜欢写 80 行的"全能函数"——读文件、处理数据、画图全塞一起。

```
✅ "把这个功能拆成 3 个小函数：read_file、clean_data、plot_curves。
   每个函数不超过 30 行，不超过 3 个参数。"
```

**为什么？** 小函数更容易看懂、更容易测试、AI 下次修改时不会动到无关逻辑。

### 习惯 3：在项目根目录维护 AGENTS.md

创建一个 `AGENTS.md` 文件，放项目的全局约定。AI（Copilot Chat / Claude / ChatGPT）在理解项目时，读到这个文件会遵守约定。

```markdown
# AGENTS.md — 项目开发约定

## 技术栈
- Python 3.12 + uv 包管理
- 数据处理：openpyxl, pandas
- 出图：matplotlib
- GUI：PySide6（如涉及）

## 代码规范
- 所有公开函数必须有类型注解：`def func(x: str) -> int:`
- 路径统一用正斜杠 `/`，相对路径以项目根为起点 `./data/`
- 中文注释，英文变量名
- 数据接口统一返回 `list[dict]`，不返回 DataFrame（除非性能需要）

## 数据约定
- 空值用 `None`，不用空字符串 `""`
- 荷载单位 kN，位移单位 mm
- 输出图片 DPI ≥ 300

## 禁止事项
- 禁止在代码中硬编码文件路径
- 禁止引入项目未使用的第三方库
- AI 生成的代码提交前必须通过三关验收（读得懂 → 零红线 → F5 跑通）
```

> 💡 你可以把这段模板复制到你的项目根目录，按实际需求修改。每次新的 AI 会话开始时，把 `AGENTS.md` 的内容贴给 AI 做上下文。

## 六、提交规范

### commit 信息注明 AI 辅助

GitHub 上别人（或者半年后的你自己）看提交记录时，需要知道哪些代码是 AI 协助写的，哪些是纯手写。这不是面子问题——AI 生成的代码出 bug 的概率不同，标注清楚方便回溯。

```bash
# ✅ 好的 commit 信息
git commit -m "feat: 锚杆数据读取模块 (AI 辅助，已验收)"
git commit -m "fix: 修复荷载值为负未拦截的 bug (AI 辅助调试)"

# ❌ 不标注，或标注了但没验收
git commit -m "feat: 数据模块"
git commit -m "fix: bug (未验收，先提交)"
```

### 禁止不验收直接 push

```
你写的代码 ──→ 三关验收 ──→ git add + commit ──→ git push
                                  ↑
AI 写的代码 ──→ 三关验收 ──→ git add + commit ──→ git push
                ↑
           没过这关就 stop
```

> ⚠️ **没有例外。** 哪怕 AI 只改了一行，也必须 F5 跑一遍。你见过"只改了一行结果整段逻辑崩了"的次数会远超预期。

## 七、常见误区清单

| 误区 | 为什么错 | 正确做法 |
|------|---------|---------|
| "AI 写的代码不报错 = 正确" | AI 代码最常见的问题是**逻辑正确但业务错误**——读错了列、漏掉了空值处理、单位搞混 | 用真实数据目测输出，不只看"没报错" |
| "贴越长的需求，AI 理解越准" | 超过 500 字的提示词，AI 会选择性忽略后面的内容 | 拆成多个小任务，一个需求一次对话 |
| "Copilot 的灰色建议直接 Tab 接受" | Copilot 会猜错变量名、用不存在的 API | 先读一遍灰色字，确认逻辑对再 Tab |
| "代码跑通了就不需要类型注解" | 一个月后你回来看自己代码，没有类型注解跟读天书一样 | 写函数的同时写类型注解，不"回头补" |
| "AI 推荐的库直接 pip install" | AI 会推荐过时版本、不兼容版本、甚至**不存在的库**（幻觉） | 先去 [pypi.org](https://pypi.org) 搜一下这个库是否存在、最新版本号是多少 |
| "报错信息直接贴给 AI 修" | AI 倾向于"修症状"而不是"修根因"——把 `try/except` 包住报错行就完事了 | 先自己看报错堆栈，找到**根源行**，再让 AI 改那一行 |
| "AI 能自己理解我的项目结构" | AI 对你的项目一无所知，除非你告诉它 | 对话开头先贴项目结构 + 相关文件路径 |
| "私有代码可以随便贴给 AI 看" | 免费版 ChatGPT / Claude / Copilot Chat 的对话可能被用于训练 | 敏感代码（客户数据、密钥、核心算法）不要贴；商业项目用企业版 API |
| "AGENTS.md 写了也没什么用" | 没写 AGENTS.md，每次都重新描述项目约定——10 次对话浪费 10 次口舌 | 5 分钟写一次，以后每次对话直接贴文件内容 |
